# 株式会社上間 レポーティング自動化 CLAUDE.md

> **本ファイルの役割**: レポート自動生成パイプライン（Python）およびClaude Codeの全ルールを定義する。デザインシステム・ワークフロー・計算ロジック・レポート仕様の統一管理ファイル。

## 0. 基本ルール

- 全てのコミュニケーション・レポート出力は**日本語**で行う
- レポートHTMLは**1ファイル完結**（外部CSS/JS依存なし、Google Fonts CDN + Chart.js CDNのみ許可）
- データの二重計上・矛盾がないよう、計算ロジックを厳守する
- 未検証データ・推定値はレポート内で明示する
- inbox/ のデータは生成後も削除しない（エビデンス保持）
- **主要生成手段はPythonパイプライン**（`run_weekly_report.py`）。AI不要で0.3秒以内にHTML完成
- **Claude Codeの役割**は「改善提案セクション（§9）の高度な分析テキスト生成」および「パイプライン改善・デバッグ」に限定

## 2. データソースと入力規則

### 2.1 基幹システム「UEMAシステム」

自社オリジナルPOSと連動した基幹システム。売上・原価・客数・ロス・人件費・棚卸し・POSデータをCSVエクスポート可能。

### 2.2 データモード（旧フォーマット / 統合フォーマット）

パイプラインは**2つのデータモード**を自動検出して切り替える:
- **旧フォーマット**: 店舗別CSV + 営業日報PDF（`extract_data.py`で処理）
- **統合フォーマット（推奨）**: 全店舗統合CSV（`extract_unified.py`で処理）

`run_weekly_report.py` が inbox/ に `pos_item_*.csv` を検出すると自動的に統合モードに切り替わる。

### 2.3 統合フォーマット（推奨・4種 + 予算Excel）

#### ① pos.csv（会計POS・全店舗統合）

| カラム | 説明 |
|-------|------|
| ID | POS売上ID |
| タイムスタンプ | 会計日時（YYYY-MM-DD HH:MM:SS） |
| 販売種別 | テイクアウト / イートイン |
| 支払方法 | 現金支払い / クレジットカード / 電子マネー 等 |
| **注文ID** | **空=前出売上、値あり=注文売上（EC経由）**。前出/注文分離の唯一の判定基準 |
| 店舗ID / 店舗名 | 全7店舗 + ドラモリ + 法人 + 配送 |
| 合計金額(税込) / 消費税 | 売上金額 |
| 性別 / 年代 | 顧客属性（ペルソナ分析用）。レジスタッフ目視入力のため入力バイアスあり |

エンコーディング: UTF-8 BOM。ファイル名: `pos_YYYYMMDD_HHMMSS.csv`

#### ② pos_item.csv（商品明細POS・全店舗統合）

| カラム | 説明 |
|-------|------|
| POS売上ID | pos.csvのIDと紐付き → バスケット分析の結合キー |
| タイムスタンプ | 商品購入日時 |
| 店舗名 | 全チャネル |
| 商品ID / 商品名 | 商品マスタ連携 |
| 数量 / 単価(税込) | 明細金額 |
| 性別 / 年代 | 顧客属性 |

エンコーディング: UTF-8 BOM。ファイル名: `pos_item_YYYYMMDD_HHMMSS.csv`

#### ③ disposal.csv（廃棄・全店舗統合）

| カラム | 説明 |
|-------|------|
| 店舗名 | 7店舗 + 工場統合 |
| 廃棄日 | YYYY-MM-DD |
| 廃棄・割引区分 | 廃棄（商品）/ 廃棄（仕掛品）/ 廃棄（原材料）/ 値引調整 / 従業員割引 |
| 商品名 / 仕掛品名 / 原材料名 | いずれか1つが記入 |
| 数量 / ロス合計（税込み） | ロス金額 |

エンコーディング: UTF-8 BOM。ファイル名: `disposal_YYYYMMDD_HHMMSS.csv`

#### ④ 人件費CSV（店舗別7ファイル）

1行目: 集計期間、2行目: ヘッダー。部門名=店舗名で自動マッチング。
カラム: 部門名 / 社員名 / 時給 / 出勤日数 / 各種時間・手当 / 合計時間 / 手当合計

エンコーディング: UTF-8 BOM

#### ⑤ 予算Excel（月次・任意）

ファイル名に「予算」を含む `.xlsx`。日別×店舗別の売上予算 + 月次人件費予算を格納。
`extract_unified.py` が `--start` / `--end` 期間内の日別予算を合算し、予算達成率・売上スコアを算出。

#### ⑥ ABC分析CSV（店舗別・粗利データ用）

pos_itemに原価列がないため、粗利分析には引き続きABC分析CSVが必要。
`extract_unified.py` がABC売上合計とPOS前出売上を比較し、全7店舗に自動マッチング。
カラム: 商品ID / 商品名 / 数量(Q) / 売上(税込) / 売上(税抜) / 単価(税抜)(P) / 原価(V) / 粗利額(M) / 総原価額(VQ) / 総粗利額(MQ)

**売上構造の必須理解:**
- **店舗前出売上**: pos.csvの**注文IDが空**の取引。店頭直売。**ロスはこのカテゴリのみで発生**
- **注文売上**: pos.csvの**注文IDが値あり**の取引。EC注文（電話/Web/店頭予約）がPOSで精算されたもの
- **合計売上** = 前出売上 + 注文売上（全てpos.csv内で完結。**ec.csvの店舗受取分はPOSと重複するため加算しない**）
- **二重計上防止**: EC注文の「店舗受取」はPOS精算済み。ec.csvは全社レポート（配達チャネル分析）用

**平日/週末の分類（`datetime` で自動判定）:**
- 週末 = 土曜・日曜
- 平日 = 月〜金（祝日は考慮しない）

### 2.4 旧フォーマット（後方互換）

統合フォーマットが検出されない場合、旧フォーマットで処理する。

- **営業日報PDF**: 店舗別×日別（pdftotext + 正規表現で解析）
- **ABC分析CSV**: 商品別集計（Shift_JIS、1行目:集計期間、2行目:ヘッダー）
- **廃棄リストCSV**: 店舗別（UTF-8-SIG）
- **POSデータCSV**: 旧フォーマット（Shift_JIS、会計日時/店舗/扱い者/売上）
- **売上実績CSV**: 時間帯付き商品別（Shift_JIS）
- **人件費CSV**: 店舗別（UTF-8-SIG）

### 2.5 inbox/ へのデータ投入ルール — 「放り込むだけ」方式

**最重要原則: バックオフィスは `inbox/` にファイルをそのまま放り込むだけ。ファイル名の命名規則・フォルダ分け・リネームは一切不要。パイプラインがファイルの中身を読み取って、店舗名・日付・データ種別を自動判定し、整理・処理する。**

#### ファイル自動判定ロジック

| データ種別 | 判定手がかり | 処理スクリプト |
|-----------|------------|-------------|
| **統合POS** | ファイル名 `pos_*.csv` | `extract_unified.py` |
| **統合POS明細** | ファイル名 `pos_item_*.csv` | `extract_unified.py` |
| **統合廃棄** | ファイル名 `disposal_*.csv` | `extract_unified.py` |
| **予算Excel** | ファイル名に「予算」+ `.xlsx` | `extract_unified.py` |
| ABC分析CSV | ファイル名に "abc" or ヘッダーに「総原価額」「総粗利額」 | 両方 |
| 人件費CSV | ファイル名に "人件費" or ヘッダーに「深夜勤時間」「手当合計」 | 両方 |
| 営業日報PDF | `.pdf` 拡張子 | `extract_data.py`（旧モード） |
| 旧POSデータCSV | ヘッダーに「会計日時」 | `extract_data.py`（旧モード） |
| 旧廃棄リストCSV | ファイル名に "廃棄" or ヘッダーに「廃棄区分」「廃棄合計」 | `extract_data.py`（旧モード） |
| 売上実績CSV | ファイル名に "売上実績" or "売上" and ヘッダーに「商品名」「売上」 | 両方 |

#### バックオフィスに求める最低限のこと

1. **UEMAシステムからCSV/PDFをエクスポートして inbox/ に保存する** — それだけ
2. ファイル名はUEMAシステムのデフォルト出力名のままでOK
3. フォルダ分けは不要（inbox/ 直下にフラットに置いてよい）
4. 複数週・複数店舗のファイルを混在させてもOK（自動仕分け）
5. 特記事項があればテキストファイル（.md or .txt）を1つ添えるだけ

#### 処理済みファイルの管理

```
inbox/
├── （未処理ファイルをここに放り込む）
└── processed/           ← run_weekly_report.py が処理済みファイルを自動移動
    └── 2026/03/W14/     ←   年/月/週でフォルダを自動作成
```

## 3. ワークフロー

### 3.1 店舗別週次レポート生成（メインフロー）

**Pythonパイプラインによる完全自動生成。全7店舗を約1.2秒で一括生成。**

```
inbox/ にファイル投入（放り込むだけ方式）
  │
  ▼ モード自動検出: pos_item_*.csv があれば統合モード、なければ旧モード
  │
  ▼ Step 1a（統合モード）: extract_unified.py              ← 約0.9秒（全7店舗一括）
  ▼ Step 1b（旧モード）: extract_data.py                   ← 約0.2秒（1店舗ずつ）
  data/{店舗名}/{年}/{月}/{週}.json
  │
  ▼ Step 2: generate_report.py（Python, 非AI）            ← 約0.3秒（7店舗分）
  reports/{店舗名}/{年}/{月}/{週}.html
  │
  ▼ Step 3（任意）: Claude Code（AI）                      ← バスケット分析セクション注入
  │
  ▼ Step 4: inbox/processed/{年}/{月}/{週}/ にファイル移動   ← 自動
```

#### フォルダ構造（店舗優先）

```
reports/{店舗名}/{年}/{月}/{週}.html    例: reports/登川店/2026/03/W12.html
data/{店舗名}/{年}/{月}/{週}.json      例: data/登川店/2026/03/W12.json
inbox/processed/{年}/{月}/{週}/        例: inbox/processed/2026/03/W12/
```

#### 実行コマンド

```bash
cd scripts/
python3 run_weekly_report.py --open                              # 全店舗生成+ブラウザ表示
python3 run_weekly_report.py --store 登川店                       # 特定店舗のみ
python3 run_weekly_report.py --start 2026-04-01 --end 2026-04-07  # 期間指定（統合モード用）
python3 run_weekly_report.py --no-move                           # inbox/のファイルを移動しない
python3 run_weekly_report.py --skip-ai                           # AI改善提案をスキップ（高速テスト用）
```

## 3.7 テンプレート方式

- `templates/weekly_store.html` がCSS/JSを固定で保持（確定版デザインシステム格納済み）
- データ部分のみ `{{PLACEHOLDER}}` 形式（大文字スネークケース）

## 4. 計算ロジック

> 以下のロジックは `scripts/utils/calculations.py` に実装済み。

### 週次計算（`calculations.py` に実装済み）

- **売上**: 合計=Σ(前出+注文)、予算達成率=合計÷予算×100
- **客数・客単価**: 客単価=前出売上÷客数
- **人件費**: 人件費率=人件費÷合計売上×100、推定労働時間=Σ(前出÷人時生産高)
- **ロス（分母は必ず前出売上）**: ロス率=ロス÷前出×100。内訳: 値引調整/廃棄/従業員割引
- **時間帯別**: 前出のみ対象、5帯(06-08/09-11/12-14/15-17/18+)、平日・週末それぞれ日平均
- **平日/週末比較**: 差分%=(週末日平均÷平日日平均−1)×100。ロス率等は色の正負反転
- **売れ筋**: TOP10、平日/週末/合計個数+売上合計（前出売上の平日/週末比率で按分）

### 週次成績スコア（100点満点、`calculations.py` に実装済み）

```
売上達成(40点): 100%→40 / 98%→34 / 95%→26 / 92%→18 / 90%→10 / <90%→0
ロス率(30点):   ≤3%→30 / ≤4%→24 / ≤5%→18 / ≤6%→12 / ≤7%→6 / >7%→0
人件費(30点):   ≤25%→30(残業なし)/24(残業あり) / 以降×0.6→0.4→0.2→0
総合ランク: S(95+) / A(80+) / B(65+) / C(50+) / D(<50)
```

### 商品分析（`calculations.py` に実装済み）

- **粗利貢献TOP20**: ABC分析CSVの `総粗利額(MQ)` を商品名でグルーピング → 粗利額降順TOP20
- **粗利貢献度**: 各商品の粗利額 ÷ 全商品粗利合計 × 100。累積%も算出
- **2×2マトリックス**: 粗利額(Y軸) × 販売個数(X軸) でTOP30商品をプロット。中央値で4象限分割（花形/問題児/金のなる木/負け犬）
- **製造ロス率**: 廃棄数量 ÷ (販売数量 + 廃棄数量) × 100。ワースト10表示。15%以上は赤ハイライト

### 注意アイテム判定（`calculations.py` `calc_alert_products` に実装済み）

**A. 廃止すべき商品:**
- 販売ゼロで廃棄あり（行事用商品は除外）
- 廃棄額 > 粗利額 かつ 販売10個未満（実質赤字）

**B. ロット削減すべき商品:**
- 製造ロス率25%以上（A判定を除く）
- 平日/週末の適正製造数 = 1日販売数 × 1.1（安全在庫10%）を具体提示

### 増産チャンス判定（`calculations.py` `calc_opportunity_products` に実装済み）

**商品カテゴリ別のオペレーション分類:**

| カテゴリ | 製造元 | 増産対応 | キーワード判定 |
|---------|--------|---------|-------------|
| 弁当・おにぎり | セントラルキッチン（工場） | 前日〜当日朝の発注で決まり、当日増産不可 | 弁当/おにぎり/パック/いなり/寿司/じゅーしー/ボロボロ/イナリ/ポーク玉子 等 |
| 天ぷら・ホットフード | 店舗製造（店頭） | 材料があれば需要に合わせ柔軟対応可 | 上記以外 |

**1. 工場発注品（弁当・おにぎり）: 15時売り切り最適発注**
- 06-14時の販売実績 = 「15時までに売り切る」最適量
- ピーク帯(10-13時)に販売0の時間帯がある = 欠品 → 発注増を提案
- 平日/週末別の最適発注数を具体提示

**2. 店舗製造品（天ぷら・ホットフード）: 高粗利ポテンシャル**
- 粗利率55%以上 かつ 販売数が上位20商品平均の半分以下
- 製造+20%の増産提案 + 粗利インパクト額を算出

**ランキング:** 全候補を粗利インパクト（増産時の週間粗利増加額）降順でソートし、上位10件のみ表示

### 月次計算

- 月次売上=該当月の全週次合算、粗利=売上−原価、営業利益=粗利−販管費
- YoY=(当月−前年同月)÷前年同月×100、MoM=(当月−前月)÷前月×100
- 進捗率=実績÷計画×100、シナリオ判定=Low/Mid/Highと比較

## 9. データ処理注意事項

1. **日付・曜日**: `datetime` で算出。手入力禁止。平日/週末の分類も曜日から自動判定
2. **数値フォーマット**: 3桁カンマ区切り、通貨は `¥` プレフィックス。`parse_number` は通貨記号（`¥` / `￥`）にも対応
3. **CSVエンコーディング**: `chardet` 等で自動判別（Shift_JIS / UTF-8-SIG 両対応）。`csv_parser.py` の `detect_encoding` に実装済み
4. **棒グラフ高さ**: 最大値を基準に比率で算出（サイズは§5.4参照）。日別チャートは `daily-chart-wrap` で横スクロール対応
5. **ロス計算の分母は前出売上**: ロス率・各区分%はすべて「店舗前出売上」を分母
6. **売上の二重計上禁止**: 合計売上=前出+注文。ロス・時間帯分析は前出ベースのみ
7. **店舗名のロゴマーク**: 店舗名から「店」を除いた文字（例: 山内店→「山内」）
8. **製造ロス率の定義**: 廃棄数量 ÷ (販売数量 + 廃棄数量) × 100。15%以上は赤ハイライト
9. **廃棄TOP10のフィルター**: 廃棄区分="廃棄" のみ対象（値引調整・従業員割引は除く）
10. **週末日数の変動**: 実際の曜日から動的算出。週末0日・平日7日のケースも正しく処理
11. **時間帯売上のPDF抽出**: pdftotextの-layout出力では、時間帯セクションの「時間帯別客数」「売上」「客単価」はラベルとデータが同一行に並ぶ形式。`pdf_parser.py`の`_extract_time_slots`はセクション位置からの相対行で判定する方式
12. **商品カテゴリ分類**: 弁当/おにぎり/パック/いなり/寿司/じゅーしー/ボロボロ/イナリ/ポーク玉子 = 工場発注品。それ以外 = 店舗製造品
13. **15時売り切り原則**: 弁当・おにぎりは15時までに売り切る前提で最適発注数を算出。ランチピーク(~13時)後の需要減は正常パターンであり欠品ではない
14. **欠品検知**: ピーク帯(10-13時)に販売0の時間帯がある工場発注品のみ検出。天ぷら・ホットフードは店舗で柔軟に増産可能なため欠品判定の対象外
15. **残業表示**: has_overtime(bool) + overtime_hours(float) で実時間を表示（「あり/なし」ではなく「X.Xh」）
16. **スコア計算（仕様統一）**: ロス率スコアは ≤3%=30 / ≤4%=24 / ≤5%=18 / ≤6%=12 / ≤7%=6 / >7%=0。Python(`calc_loss_score`)とJS(`calcLossScore`)で同一基準を使用すること
17. **前出/注文分離（統合モード）**: pos.csvの`注文ID`列で判定。空=前出売上、値あり=注文売上。EC注文の「店舗受取」はPOS精算済みのため、ec.csvの店舗受取分は店舗売上に加算しない（二重計上防止）
18. **ABCマッチング（統合モード）**: ABC分析CSV = 前出売上のみの商品集計。pos.csvの注文ID空の売上合計と比較してマッチング。全7店舗で差額0〜数十円の精度
19. **バスケット分析**: pos.csvとpos_item.csvをPOS売上IDで結合。前出のみ（注文ID空）。レジ袋除外。併買ペアは`itertools.combinations`で全組み合わせをカウント
20. **顧客属性の入力バイアス**: 性別・年代はレジスタッフ目視入力。70代男性過多・男性比80%等の偏りが確認されている。ペルソナ分析は「相対的な傾向把握」用途として参照し、絶対値としては扱わない
21. **予算データ**: 予算Excelの日別×店舗別売上を`--start`/`--end`期間で合算。予算なし時の売上スコアは暫定24点

### 天候データ連携仕様

#### データソース
Open-Meteo API（無料・登録不要）で沖縄市（lat=26.3344, lon=127.8056）の過去天気を自動取得。
```
https://api.open-meteo.com/v1/forecast?latitude=26.3344&longitude=127.8056
&daily=weather_code,temperature_2m_max,temperature_2m_min,precipitation_sum,wind_speed_10m_max
&timezone=Asia/Tokyo&start_date={開始日}&end_date={終了日}
```

#### WMOコード → 天気分類

| 分類 | 条件 | アイコン | 日別チャートでの表示 |
|------|------|---------|-------------------|
| `sunny` | code ≤ 1 かつ precip < 1mm | ☀ | 曜日ラベル下に小アイコン |
| `cloudy` | code 2-3 かつ precip < 1mm | ☁ | 同上 |
| `light_rain` | precip 1-10mm | 🌧 | 同上 + 棒グラフ背景を薄い青 |
| `rain` | precip ≥ 10mm | ⛈ | 同上 + 棒グラフ背景を薄い青 + ⚠バッジ |

#### JSON daily配列への天候フィールド追加

```json
{
  "date": "2026-04-01",
  "weather": {
    "code": 96,
    "label": "雷雨",
    "icon": "⛈",
    "class": "rain",
    "precipitation_mm": 18.7,
    "temp_max": 23.8,
    "wind_max_kmh": 28.6
  },
  ...
}
```

#### 日別売上チャートへの天候表示
- 各日の曜日ラベルの上に天気アイコンを表示
- `rain`（≥10mm）の日は棒グラフの背景に`rgba(34,102,170,0.08)`の薄い青を敷く
- ツールチップに降水量・最高気温を表示

#### ロス分析セクションへの天候相関アラート
- 雨の日（≥10mm）と晴/曇の日のロス率を比較
- **雨の日のロス率が晴/曇より1.5pt以上高い場合**、以下のアラートを廃棄ロス分析セクション内に表示:

```
⚠ 天候アラート: 雨の日のロス率 X.X% は晴/曇の日（Y.Y%）より +Z.Zpt 高い傾向です。
雨予報の日は天ぷらの製造量を 15-20% 削減することを推奨します。
```

- 1.5pt未満の差の場合はアラート非表示（データ蓄積を待つ）
- 清明祭・旧盆等の行事日（`meta.event_flags`）は相関計算から除外する（将来実装）
